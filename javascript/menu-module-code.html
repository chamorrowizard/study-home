Dynamic UI & State Architecture

An in-depth code study guide on item rendering, memory tracking, and DOM lifecycle events.

Global State Layer

The architecture begins by initializing a data store outside the lifecycle of the UI rendering engine.

// Global session state memory tracking card orders const orderSelections = {};

Why this approach?

By declaring orderSelections as a global constant literal object, the application decouples data storage from DOM structures. If the menu category changes and the screen clears, the cart counts persist securely in application memory.

The Clear-and-Render Pipeline

Inside the main loadMenuModule function, the target container is retrieved and dynamically purged.

const gridTarget = document.getElementById("menu-cards-target"); gridTarget.innerHTML = "";

Why this approach?

Setting innerHTML = "" clears all previously built card structures out of the view layer instantly. This creates a clean slate prior to traversing the array to ensure no duplicate card UI blocks stack up when switching item groupings.

Defensive Design & Assets Mapping

Every dataset loop processes local fallbacks and extracts targeted assets dynamically via key lookups.

const currentQty = orderSelections[item.name] || 0; const itemDesc = item.desc ? item.desc : "Available for full scale event booking..."; // Fallback gracefully to hide the image block entirely if no image is mapped yet const imageSrc = imageMap[item.name]; const mediaHTML = (imageSrc && imageSrc.trim() !== "") ? `<img src="images/${imageSrc}" ...>` : `<div class="img-placeholder" style="display:none;"></div>`;

Why this approach?

  • The || 0 guard safely parses properties that have not been written to memory yet, transforming undefined data straight into a clean, displayable baseline digit.
  • The ternary conditional statements check for missing imagery strings. Instead of rendering empty broken graphic tags, it cleanly isolates code blocks, outputting hidden structures to maintain clean visual aesthetic continuity throughout production.

Event Propagation Mechanics

To avoid nested interface triggers layout conflicts, event bubbling must be interceptively handled within the quantitative control sub-components.

`<div class="quantity-controller" onclick="event.stopPropagation();">`

Why this approach?

The card container features its own distinct click activation event handler. Without incorporating event.stopPropagation() directly into the inner control element, adjusting the quantity values upwards or downwards via the step-buttons would simultaneously bubble upwards to fire the outer card wrapper container trigger, resulting in layout actions conflicting.

The Mutation Bridge (UI to State Connection)

The code uses a deliberate split pattern: listening to interface inputs, transforming values, and updating through a specialized driver function.

// Card Click Mutation Event card.addEventListener("click", () => { if ((orderSelections[item.name] || 0) === 0) { updateDirectQuantity(item.name, 1, inputField); } }); // Direct Field Typing Event Input Listener inputField.addEventListener("input", (e) => { let val = parseInt(e.target.value, 10); if (isNaN(val) || val <= 0) val = 0; updateDirectQuantity(item.name, val, inputField); });

Why this approach?

Instead of scattering isolated value manipulation mutations inline inside four different structural places (card click, minus tap, plus tap, field input changes), all event listeners pass the values downstream into a centralized worker method named updateDirectQuantity(). This keeps data operations predictable, dry, and highly scalable.

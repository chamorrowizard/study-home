Form Basics Cheat Sheet

Form Anatomy

The container that defines how and where data is sent.

Base Structure (Action & Method) 
<form action="/api/submit" method="POST">
  <!-- inputs go here -->
</form>
Action: Destination URL.
Method: GET (retrieving) or POST (sending/sensitive data).

Input Versatility

Text, Password & Email 
<label for="email">Email</label>
<input type="email" id="email" name="user_email" placeholder="dev@example.com">
Numbers & Dates 
<input type="number" name="qty" min="1" max="10">
<input type="date" name="birthday">

Selection Logic

Radio (Single) vs Checkbox (Multiple) 
<!-- Radios must share the same 'name' to group -->
<input type="radio" name="size" value="sm" id="s"> <label for="s">Small</label>
<input type="checkbox" name="terms" id="t"> <label for="t">Accept</label>

Senior Dev Tips

Accessibility First: Always associate a <label> with an id. It increases the click target size and is vital for screen readers.

The 'Name' Attribute: Without the name attribute, data will not be sent to the server. Treat the name as your key in a JSON object.
Global Form Attributes 
<form 
  autocomplete="on" 
  novalidate 
  target="_blank" 
  enctype="multipart/form-data">
  <!-- novalidate: bypasses browser validation -->
  <!-- enctype: required for file uploads -->
</form>

Autocomplete: Helps users fill forms faster.

Novalidate: Useful when you want to handle all validation via JavaScript.

Specialized Controls (Range, Tel, Search) 
<!-- Volume/Range Slider -->
<input type="range" min="0" max="100" step="10">

<!-- Telephone (triggers numeric pad on mobile) -->
<input type="tel" pattern="[0-9]{3}-[0-9]{3}-[0-9]{4}">

<!-- Search (includes a clear 'X' button) -->
<input type="search" placeholder="Search...">
The Accessibility Gold Standard 
<!-- Method A: Explicit Association (Best Practice) -->
<label for="user-id">Username</label>
<input id="user-id" type="text" aria-describedby="hint">
<small id="hint">Must be 8+ characters.</small>

<!-- Method B: Implicit Nesting -->
<label>
  Email:
  <input type="email">
</label>
Pro Tip: Use aria-describedby to link help text to an input so screen readers read the instructions automatically.

Multi-line Text & Specialized Types 
<!-- Textarea for longer responses -->
<label for="bio">Biography</label>
<textarea id="bio" name="bio" rows="5" cols="33">Initial content...</textarea>

<!-- Password masking -->
<input type="password" id="pass" name="password">
Grouped Radio & Multi-select Checkboxes 
<!-- Radio Group: Must share 'name' to ensure single choice -->
<input type="radio" id="r1" name="choice" value="1" checked>
<label for="r1">Option 1</label>

<!-- Checkbox: Allows multiple independent choices -->
<input type="checkbox" id="c1" name="interest" value="coding">
<label for="c1">Coding</label>
Radio (Single): Yes No

Checkbox (Multiple): Subscribe Notify
The Three Button Types 
<!-- Submits the form (Default behavior) -->
<button type="submit">Submit Form</button>

<!-- Clears all form data -->
<button type="reset">Reset Fields</button>

<!-- Does nothing by default (Use with JS) -->
<button type="button">Generic Action</button>
Data Identifiers: The name attribute acts as the variable name for the data when it reaches the server. Without a name, the input is ignored during submission.

GET vs POST: Use GET for retrieving data (like search results) where parameters appear in the URL. Use POST for sensitive data or actions that change the server state (like payments).

Validation & UI Feedback

Constraint Validation CSS 
/* Style inputs based on validity state */
input:invalid {
  border-color: #ff4d4d;
}

input:focus:invalid {
  outline: none;
  box-shadow: 0 0 0 3px rgba(255, 77, 77, 0.2);
}

/* Show feedback only after user interaction */
input:user-invalid + .error-message {
  display: block;
}
Note: Browsers [validate email formats](https://www.theodinproject.com/lessons/node-path-intermediate-html-and-css-form-basics#the-type-attribute) and required fields automatically.

The GET vs POST Rule:
  • GET: Used to *retrieve* data. Parameters are visible in the URL (e.g., Google Search).
  • POST: Used to *change* data on the server (e.g., payments or account creation). It is more secure for sensitive info.
The Payload Object: When you submit a form, the browser creates a "form" object where the name attribute is the key and the user input is the value. 
"form": {
  "first_name": "John",
  "age": "33"
}
